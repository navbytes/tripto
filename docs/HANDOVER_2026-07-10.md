## The production bug (#6) — worth understanding

Postgres re-checks a table's **SELECT** policy on `INSERT ... RETURNING`,
not just the INSERT policy. `trips`' `AFTER INSERT` trigger (which grants
the creator membership via `trip_members`, which is what `trips_select`'s
`is_trip_member(id)` check depends on) fires *after* that RETURNING check —
so a trip's own creator failed RLS on any insert that asked for the row
back, which is exactly what `supabase-swift`'s `.insert().select()` does.
Every new trip creation was silently broken in production until this was
fixed. Root-caused via live curl repro + direct SQL simulation, confirmed
fixed with a before/after live test (`403` → `201` with full row body).
Fix: `trips_select` now also allows `created_by = auth.uid()` directly, not
just `is_trip_member(id)`. Saved to memory as a pattern to watch for
elsewhere (any future "insert row, trigger grants visibility" design).

## AI Gateway (#10) — three bugs found fixing this live

1. **Wrong Anthropic model ID.** Default was `anthropic/claude-haiku-4-5`
   (hyphens); AI Gateway's catalog uses `anthropic/claude-haiku-4.5` (period
   before the version) — 404 on every call.
2. **OpenAI-shaped tool-calling doesn't round-trip for `anthropic/*` models**
   on Cloudflare's endpoints. Confirmed live: the request needs
   Anthropic-native `tools`/`tool_choice` shapes to even be accepted, and
   then the response's tool-call content isn't translated back into
   `tool_calls` at all — silently dropped. `openai/*` models round-trip
   correctly with the exact same OpenAI-shaped request our code already
   sends. Fixed by switching `DEFAULT_LLM_MODEL` to `openai/gpt-4.1-mini`
   instead of adding provider-specific branching.
3. **Wrong endpoint for scoped API tokens.** `callLLM()` called
   `api.cloudflare.com/client/v4/accounts/{id}/ai/v1/chat/completions` (the
   newer unified endpoint, ~6 weeks old as of this fix). A fine-grained
   `AI Gateway: Run` API token gets a flat `401 {code:10000}` there —
   reproduced with four separately-created tokens, one of which
   independently verified as valid/active via `/user/tokens/verify`. The
   identical token works immediately against the older
   `gateway.ai.cloudflare.com/v1/{account}/default/compat/chat/completions`
   endpoint. Broad account-owner OAuth sessions (`wrangler auth token`)
   don't hit this, which made it easy to miss while testing locally — only
   scoped tokens are affected. Switched `callLLM()` to the older endpoint.
   Revisit switching back if Cloudflare fixes scoped-token auth on the
   newer one.

All three fixed in `functions/_shared/extract.ts`, deployed, and verified
live: both `kind=booking` (multi-item extraction, 2 items from one paste)
and `kind=packing` tested end-to-end against production with a real
anonymous test user + trip, LLM call, and DB insert.
